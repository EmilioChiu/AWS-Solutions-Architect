Cloud Computing & AWS Global Infrastructure

🎯 ¿Por qué esto importa?

Eres backend developer. Ya sabes escribir APIs, manejar bases de datos y desplegar lógica de negocio. Pero todo ese código necesita un lugar donde ejecutarse. AWS es ese lugar — pero no es un datacenter tradicional.

AWS es infraestructura programable: cada servidor, firewall, balanceador y base de datos se controla mediante APIs. Como backend dev, esto significa que puedes tratar la infraestructura como código — igual que tratas tu lógica de negocio.

  • ✅ Describir la jerarquía global de AWS: Regions → AZs → Edge Locations.
  • ✅ Tener tu propia cuenta AWS Free Tier lista y asegurada con MFA.

☁️ 1. Cloud Computing

Cloud computing es la entrega bajo demanda de recursos de TI (cómputo, almacenamiento, bases de datos, redes) a través de internet, con pago por uso.

Antes, para lanzar un servidor web necesitabas: comprar hardware, instalarlo en un rack, configurar red, esperar semanas. Con AWS: un solo comando o clic y tienes un servidor corriendo en minutos.

Tres modelos de servicio

ModeloQué gestionas túQué gestiona AWSEjemplo
IaaS SO, app, datos Servidores, red, datacenter EC2 — alquilas una VM
PaaS Solo tu app y datos SO, runtime, escalado, parches Elastic Beanstalk — subes tu código
SaaS Solo config Todo lo demás (app completa) Gmail, Salesforce — usas el software
🧠 Para el backend dev: Piensa en IaaS como "alquilar un VPS", PaaS como "subir tu código a Heroku", y SaaS como "usar Stripe". AWS te da los tres — pero los arquitectos pasan más tiempo en IaaS y PaaS.

Ventajas clave del cloud

  • Elasticidad: Escalas de 1 a 1000 servidores en minutos.
  • Pago por uso: Pagas solo por lo que consumes, sin compromisos.
  • Global en segundos: Despliegas en cualquier región del mundo.
  • Stop guessing capacity: No necesitas adivinar cuánta capacidad necesitas.

🌍 2. Infraestructura Global de AWS

AWS no es un datacenter gigante. Es una red global de datacenters agrupados en tres niveles:

GLOBAL AWS ┌─────────────────────────────────────────────┐ │ REGION (ej: us-east-1, eu-west-1) │ │ ├── Availability Zone A (datacenter(s)) │ │ ├── Availability Zone B (datacenter(s)) │ │ ├── Availability Zone C (datacenter(s)) │ │ └── Edge Locations (CloudFront caches) │ └─────────────────────────────────────────────┘

Region

Área geográfica que contiene 2 o más Availability Zones. Cada región es completamente independiente: los datos no cruzan regiones a menos que tú lo configures explícitamente. Ejemplos: us-east-1 (N. Virginia), eu-west-1 (Irlanda), ap-southeast-2 (Sídney).

Availability Zone (AZ)

Uno o más datacenters discretos dentro de una región. Cada AZ tiene energía, redes y conectividad independientes. Están aislados entre sí (si un terremoto golpea una AZ, las otras no se ven afectadas) pero conectados con fibra de alta velocidad y baja latencia (<2ms).

💡 Clave para el examen: "Multi-AZ" es el mínimo para alta disponibilidad. Una app desplegada en una sola AZ puede caer si esa AZ falla. Dos AZs = tolerancia a fallos.

Edge Location

Son sitios de borde de CloudFront (CDN) que cachean contenido estático y dinámico cerca del usuario. Hay ~600+ Edge Locations globales. No son AZs — solo cachean datos, no ejecutan cómputo general.

¿Cómo elegir una región?

  1. Latencia: La más cercana a tus usuarios.
  2. Cumplimiento: Datos deben residir en ciertos países (GDPR).
  3. Costo: No todas las regiones cuestan igual.
  4. Servicios disponibles: No todos los servicios están en todas las regiones.

🛠️ 3. Laboratorio: Tu Primera Cuenta AWS

Vamos a poner manos a la obra. Sigue estos pasos:

  1. Crear cuenta AWS Free Tier
    Ve a aws.amazon.com/free y haz clic en "Crear una cuenta AWS".
    • Usa tu email personal (no corporativo — esto es para estudio).
    • La cuenta root requiere: email + contraseña + datos de tarjeta de crédito (para verificar identidad).
    • No te preocupes: La Free Tier incluye 750h/mes de EC2 t2.micro, 5GB S3, y más durante 12 meses.
    • Se te cobrará una verificación temporal (~$1-2 USD) que se reembolsa.
    • Si ya tienes cuenta, salta este paso.
  2. Habilitar MFA en la cuenta root
    Busca "IAM" en la consola → "Activar MFA en tu cuenta root" → "Assign MFA device".
    • Usa Google Authenticator, Authy, o 1Password como app TOTP.
    • Escanea el código QR con tu app.
    • ¡Nunca más uses la cuenta root para nada que no sea facturación o cambios críticos!
    ⚠️ Seguridad: Sin MFA, tu cuenta es vulnerable. Es el paso más importante de seguridad en AWS.
  3. Crear un usuario IAM administrador
    IAM → Users → Create user → Nombre: admin → Attach policy: AdministratorAccess.
    • Habilita "Provide user access to the AWS Management Console".
    • Establece una contraseña temporal.
    • Al primer login, te pedirá crear una nueva contraseña + habilitar MFA (¡hazlo!).
    • Usa este usuario para todo tu trabajo diario, no el root.
  4. Explorar la consola
    En la barra superior derecha, encuentra el selector de regiones.
    • Cambia entre US East (N. Virginia) y EU (Ireland).
    • Busca "EC2" en la barra de búsqueda — ve la pantalla de lanzamiento de instancias (sin crear nada).
    • Busca "S3" — ve la pantalla de buckets.
    • Busca "VPC" — ve el panel de VPCs. ¿Cuántas VPCs ves? (pista: hay una default).
  5. Instalar AWS CLI
    curl "https://awscli.amazonaws.com/AWSCLIV2.pkg" -o "AWSCLIV2.pkg"
sudo installer -pkg AWSCLIV2.pkg -target /
aws --version

    Luego configura tu perfil:

    aws configure
AWS Access Key ID: [crea una en IAM → Security credentials → Create access key]
AWS Secret Access Key: [copia secreta]
Default region name: us-east-1
Default output format: json

    Prueba que funciona:

    aws ec2 describe-regions --output table

    ¡Deberías ver una tabla con todas las regiones AWS!

🧪 4. Mini-Quiz

Responde estas preguntas para verificar lo aprendido.

1. ¿Cuántas Availability Zones debe tener una región como mínimo?

✅ 2 AZs — Cada región tiene al menos 2 AZs (algunas hasta 6). Esto permite alta disponibilidad Multi-AZ.

2. Como backend developer, ¿cuál modelo de cloud te da más control sobre el SO y runtime?

✅ IaaS — Con IaaS (EC2) tienes control total del SO, runtime, librerías y configuración. Es como alquilar un servidor dedicado.

3. ¿Qué usas para interactuar con AWS desde tu terminal?

✅ AWS CLI — El CLI te permite ejecutar comandos AWS desde tu terminal. Los SDKs son para integrar AWS en tu código.

4. ¿Cuál es la diferencia principal entre una Region y un Edge Location?

✅ Las Regions ejecutan servicios de cómputo y almacenamiento. Los Edge Locations son solo puntos de cacheo de CloudFront para servir contenido cerca del usuario.

5. ¿Por qué deberías crear un usuario IAM en lugar de usar la cuenta root?

✅ La cuenta root tiene acceso total y sin restricciones. Por seguridad, se recomienda crear usuarios IAM con permisos limitados (least privilege) para el trabajo diario.
